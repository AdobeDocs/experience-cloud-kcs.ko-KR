# 캐싱 이해 | AEM


이 문서에서는 기본 설치에서 다음 기본 캐시 디렉터리를 사용하여 Dispatcher 캐싱을 수행하는 방법과 이를 구성하는 방법에 대해 알아봅니다.

- 작성자

   - /mnt/var/www/author
- 게시자

   - /mnt/var/www/html


## 설명 {#description}


### <b>환경</b>

Adobe Experience Manager

### <b>문제/증상</b>

이 문서에서는 Dispatcher 캐싱 발생 방법 및 구성 방법을 설명합니다.
<br><br> <br>

## 해결 방법 {#resolution}


<b>캐싱 디렉터리</b>

기본 설치에서는 다음과 같은 기본 캐시 디렉터리를 사용합니다

- 작성자

   - /mnt/var/www/author
- 게시자

   - /mnt/var/www/html


각 요청이 Dispatcher를 통과할 때 요청은 구성된 규칙을 따라 적격 항목의 응답에 대해 로컬로 캐시된 버전을 유지합니다.

<b>참고:</b>

Apache가 DocumentRoot에서 파일을 찾을 때 어떤 AEM 인스턴스에서 발생했는지 모르므로, 게시된 워크로드를 작성자 워크로드와 의도적으로 분리시킵니다. 따라서 작성자 팜에 캐시가 비활성화되어 있더라도 작성자의 DocumentRoot가 게시자와 동일하면 존재하는 경우 캐시에서 파일을 제공합니다. 즉, 게시된 캐시에서 작성자 파일을 제공하고 방문자를 위한 매우 끔찍한 믹스 매치 경험을 만듭니다. 서로 다른 게시된 콘텐츠에 대해 별도의 DocumentRoot 디렉터리를 유지하는 것도 매우 잘못된 생각입니다. Clientlib과 같이 사이트마다 차이가 없는 캐시된 항목을 여러 개 다시 만들어야 할 뿐만 아니라 설정한 각 DocumentRoot에 대해 복제 플러시 에이전트를 설정해야 하므로 각 페이지 활성화로 플러시 오버헤드의 양이 늘어납니다. 파일 네임스페이스 및 전체 캐시 경로를 사용하고 게시된 사이트에 대해 여러 DocumentRoots를 사용하지 마십시오.



<b>구성 파일</b>

Dispatcher는 팜 파일의 /cache 섹션에서 캐시 가능한 대상을 제어합니다. 
AMS 기본 구성 팜에서는 아래와 같이 의 포함을 찾을 수 있습니다.


```
/cache { 
    /rules { 
        $include "/etc/httpd/conf.dispatcher.d/cache/ams_author_cache.any" 
    }
```


캐시할 내용 또는 캐시하지 않을 내용에 대한 규칙을 생성할 때 설명서를 참조하십시오 [여기](https://experienceleague.adobe.com/docs/experience-manager-dispatcher/using/configuring/dispatcher-configuration.html?lang=en#configuring-the-dispatcher-cache-cache)



<b>캐싱 작성자</b>

사람들이 작성자 콘텐츠를 캐시하지 않는 곳에서 확인한 구현은 많습니다. 
이들은 작성자에 대한 성능과 응답성에 있어서 엄청난 업그레이드를 놓치고 있습니다.

제대로 캐시하기 위해 작성자 팜을 구성하는 전략에 대해 설명하겠습니다.

다음은 작성자 팜 파일의 기본 작성자/캐시 섹션입니다.


```
/cache { 
    /docroot "/mnt/var/www/author" 
    /statfileslevel "2" 
    /allowAuthorized "1" 
    /rules { 
        $include "/etc/httpd/conf.dispatcher.d/cache/ams_author_cache.any" 
    } 
    /invalidate { 
        /0000 { 
            /glob "*" 
            /type "allow" 
        } 
    } 
    /allowedClients { 
        /0000 { 
            /glob "*.*.*.*" 
            /type "deny" 
        } 
        $include "/etc/httpd/conf.dispatcher.d/cache/ams_author_invalidate_allowed.any" 
    } 
}
```


여기서 주목할 중요한 사항은 <b>/docroot</b> 작성자의 캐시 디렉터리로 설정됩니다.

<b>참고:</b>

작성자의 .vhost 파일에 있는 DocumentRoot 가 farms /docroot 매개 변수와 일치하는지 확인합니다.

캐시 규칙에는 파일이 포함됩니다 <b>/etc/httpd/conf.dispatcher.d/cache/ams_author_cache.any</b> 에는 다음 규칙이 포함됩니다.


```
/0000 { 
 /glob "*" 
 /type "deny" 
} 
/0001 { 
 /glob "/libs/*" 
 /type "allow" 
} 
/0002 { 
 /glob "/libs/*.html" 
 /type "deny" 
} 
/0003 { 
 /glob "/libs/granite/csrf/token.json" 
 /type "deny" 
} 
/0004 { 
 /glob "/apps/*" 
 /type "allow" 
} 
/0005 { 
 /glob "/apps/*.html" 
 /type "deny" 
} 
/0006 { 
 /glob "/libs/cq/core/content/welcome.*" 
 /type "deny" 
}
```


작성자 시나리오에서 콘텐츠는 항상 그리고 의도적으로 변경됩니다. 자주 변경되지 않는 항목만 캐시합니다.
/libs는 기본 AEM 설치의 일부이며 서비스 팩, 누적 수정 팩, 업그레이드 또는 핫픽스를 설치하기 전까지 변경될 수 있으므로 /libs를 캐시할 규칙이 있습니다. 이러한 요소를 캐시하는 것은 매우 합리적이며, 실제로 사이트를 사용하는 최종 사용자의 작성 환경에 도움이 됩니다.

<b>참고:</b>

이러한 규칙은 캐시됩니다 <b>/apps</b> 사용자 정의 애플리케이션 코드가 있는 위치입니다. 이 인스턴스에서 코드를 개발 중인 경우 파일을 저장할 때 매우 혼동되고 캐시된 복사본을 제공하기 때문에 UI에 반영되는지 여부가 표시되지 않습니다. 여기서의 목적은 AEM에 코드를 배포하는 경우 이 작업도 드물게 수행되며 배포 단계의 일부는 작성자 캐시를 삭제하는 것입니다. 캐시 가능 코드가 최종 사용자에게 더 빨리 실행될 수 있도록 하는 큰 이점도 있습니다.



<b>ServeOnStale(예: Stale/SOS에 제공)</b>

이는 Dispatcher 기능의 보석 중 하나입니다. 게시자가 로드 중이거나 응답하지 않으면 일반적으로 502 또는 503 http 응답 코드가 발생합니다. 이 경우 이 기능이 활성화되면 Dispatcher는 새 복사본이 아니더라도 최선의 노력으로 캐시에 있는 모든 콘텐츠를 제공하라는 지침을 받게 됩니다. 기능을 제공하지 않는 오류 메시지를 표시하는 것보다 기능이 있는 경우 제공하는 것이 좋습니다.

<b>참고:</b>

게시자 렌더러에 소켓 제한 시간 또는 500 오류 메시지가 있는 경우 이 기능이 트리거되지 않습니다. AEM에 연결할 수 없는 경우 이 기능은 아무 작업도 하지 않습니다.

이 설정은 모든 팜에 설정할 수 있지만, 게시된 팜 파일에만 적용하는 것은 적절합니다. 다음은 팜 파일에서 활성화된 기능의 구문 예입니다.


```
/cache { 
    /serveStaleOnError "1"
```




<b>쿼리 매개 변수/인수를 사용하여 페이지 캐싱</b>

<b>참고:</b>

Dispatcher 모듈의 일반적인 동작 중 하나는 요청에 URI(일반적으로 /content/page.html과 같이 표시됨)에 쿼리 매개 변수가 있는 경우입니다.<b>?myquery=value</b>) 파일 캐싱을 건너뛰고 AEM 인스턴스로 바로 이동합니다. 이는 이 요청을 동적 페이지로 간주하므로 캐시해서는 안 됩니다. 이는 캐시 효율성에 악영향을 미칠 수 있다.

주소 라인에 GET 인수/쿼리 매개 변수를 사용하는 AEM의 페이지가 있어 페이지의 운영을 돕지만 다른 HTML을 렌더링하지 않는 경우 이 구성 요소에 좋은 후보입니다.

페이지를 무시하고 계속 캐시할 인수를 Dispatcher에 알려 줄 수 있습니다.

예를 들어, 누군가가 URI의 인수 참조를 사용하여 그 사람이 어디에서 왔는지를 아는 소셜 미디어 딥링크 참조 메커니즘을 구축했습니다.

<b>사용 예:</b>

[https://www.retail.com/home.html?reference=android](https://www.retail.com/home.html?reference=android)

[https://www.retail.com/home.html?reference=facebook](https://www.retail.com/home.html?reference=facebook)

페이지는 100% 캐시할 수 있지만 인수가 있으므로 캐시하지 않습니다. 
이를 해결하기 위해 팜 구성 파일에 다음 섹션을 추가합니다.


```
/cache { 
    /ignoreUrlParams { 
        /0001 { /glob "*" /type "deny" } 
        /0002 { /glob "reference" /type "allow" } 
    }
```


이제 Dispatcher가 요청을 볼 때 요청에 참조라는 쿼리 매개 변수가 있고 여전히 페이지를 캐시한다는 사실을 무시합니다.



<b>응답 헤더 캐싱</b>

Dispatcher가 .html 페이지 및 clientlib을 캐시한다는 것은 거의 확실하지만, 이름이 같지만 확장명이 .h인 파일의 콘텐츠와 함께 특정 응답 헤더를 캐시할 수 있다는 사실을 알고 계십니까? 이렇게 하면 콘텐츠뿐만 아니라 캐시에서 이 콘텐츠와 함께 제공되어야 하는 응답 헤더에 다음 응답이 가능합니다.

AEM은 UTF-8 인코딩 이상을 처리할 수 있습니다.

경우에 따라 항목에 캐시 TTL의 인코딩 세부 사항과 마지막으로 수정된 타임스탬프를 제어하는 데 도움이 되는 특수 헤더가 있습니다.

이러한 값은 캐시될 때 기본적으로 제거되고, apache httpd 웹 서버는 일반적인 파일 처리 방법으로 자산을 처리하는 자체 작업을 수행하는데, 이는 일반적으로 파일 확장명에 따른 mime 유형 추측으로 제한됩니다.

디스패처가 에셋 및 원하는 헤더를 캐시하도록 한 경우 적절한 경험을 노출하고 모든 세부 정보가 클라이언트의 브라우저에 적용되는지 확인할 수 있습니다.

다음은 캐시할 헤더가 지정된 팜의 예입니다.


```
/cache { 
 /headers { 
  "Cache-Control" 
  "Content-Disposition" 
  "Content-Type" 
  "Expires" 
  "Last-Modified" 
  "X-Content-Type-Options" 
 } 
}
```


이 예제에서는 CDN이 캐시를 무효화할 시기를 알 수 있도록 헤더를 제공하도록 AEM을 구성했습니다. 즉, 이제 AEM에서 헤더에 따라 무효화되는 파일을 적절히 지시할 수 있습니다.

<b>참고:</b>

정규 표현식이나 glob 일치를 사용할 수 없습니다. 캐시할 머리글의 리터럴 목록입니다. 캐시할 리터럴 헤더 목록에만 넣습니다.



<b>유예 기간 자동 무효화</b>

페이지 활성화가 많은 작성자의 활동이 많은 AEM 시스템에서는 반복 무효화가 발생하는 경합 조건이 있을 수 있습니다. 많이 반복되는 플러시 요청은 불필요하며, 유예 기간이 지워질 때까지 플러시를 반복하지 않도록 일부 허용치에 빌드할 수 있습니다.

<b>작동 방식의 예:</b>

/content/exampleeco/en/을 무효화하는 요청이 5개 있는 경우 모두 3초 이내에 발생합니다.

이 기능을 사용하면 캐시 디렉터리 /content/exampleeco/en/을 5번 무효화합니다.

이 기능을 켜고 5초로 설정하면 캐시 디렉터리 /content/exampleco/en/once가 무효화됩니다.

다음은 5초 유예 기간 동안 구성되는 이 기능의 구문 예제입니다.


```
/cache { 
    /gracePeriod "5"
```




<b>TTL 기반 무효화</b>

Dispatcher 모듈의 새로운 기능은 다음과 같습니다. <b>TTL(Time to Live)</b> 캐시된 항목에 대한 기반 무효화 옵션. 항목이 캐시되면 캐시 제어 헤더가 있는지 검색하고 캐시 디렉터리에 동일한 이름과 <b>.ttl</b> 확장명.

다음은 팜 구성 파일에 구성된 기능의 예입니다.


```
/cache { 
    /enableTTL "1"
```


<b>참고:</b>

Dispatcher가 TTL 헤더를 따르도록 AEM을 계속 구성해야 합니다. 이 기능을 전환하면 Dispatcher는 AEM이 캐시 제어 헤더를 보낸 파일을 제거할 시기만 알 수 있습니다. AEM이 TTL 헤더 전송을 시작하지 않는 경우 Dispatcher는 여기에서 특별한 작업을 수행하지 않습니다.



<b>캐시 필터 규칙</b>

다음은 게시자에서 캐시할 요소에 대한 기준 구성의 예입니다.


```
/cache{ 
    /0000 { 
        /glob "*" 
        /type "allow" 
    } 
    /0001 { 
        /glob "/libs/granite/csrf/token.json" 
        /type "deny" 
    }
```


게시된 사이트를 가능한 한 탐욕스럽게 만들고 모든 것을 캐시하려고 합니다.

캐시될 때 경험을 중단하는 요소가 있는 경우 규칙을 추가하여 해당 항목을 캐시하는 옵션을 제거할 수 있습니다. 위의 예에서 보듯이 csrf 토큰은 캐시되어서는 안 되며 제외되었습니다. 이러한 규칙 작성에 대한 자세한 내용은 을 참조하십시오 [여기](https://experienceleague.adobe.com/docs/experience-manager-dispatcher/using/configuring/dispatcher-configuration.html?lang=en#configuring-the-dispatcher-cache-cache).
