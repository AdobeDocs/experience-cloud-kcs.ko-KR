# 일반적인 중요한 AEM 문제

## 설명 {#description}


이 문서에서는 가장 일반적인 중요한 AEM 문제와 이러한 문제를 분석하는 방법에 대해 설명합니다.

- AEM Sites 성능
- AEM Assets 성능
- 메모리 문제
- 색인 지정 문제
- 복제 문제
- TarMK 손상 문제



## 해결 방법 {#resolution}

<b><u>AEM Sites 성능 문제</u></b>
 <br>
<b>성능 문제의 증상</b>

1. 페이지 로드 느림
2. 페이지 만들기 또는 편집 속도가 느림
3. AEM 응답 시간이 느립니다
4. AEM이 일부 요청에 응답하지 않습니다
5. AEM의 request.log에 응답 시간이 느립니다


<b>성능 문제의 원인</b>

1. 스레드 경합 - 느린 검색, 쓰기 중심의 백그라운드 작업, 사이트 컨텐츠의 전체 분기 이동 등과 같은 장기 실행 요청.
2. 높은 CPU 사용률
3. 고가의 검색 또는 비효율적인 애플리케이션 코드, 구성 요소 등과 같은 값비싼 요청
4. 적절한 유지 관리 부족
5. 디스패처 캐싱이 부족합니다.
6. CDN 부족
7. 브라우저 캐싱 부족
8. 페이지에 로드되고 페이지 맨 위에 로드된 스크립트가 너무 많습니다.
9. CSS가 HTML 헤드가 아니라 페이지 전체에 로드되었습니다
10. 서버 크기 조정 부족 또는 잘못된 아키텍처
11. 메모리 문제(아래 참조)


<b>성능 문제를 분석하는 방법</b>

1. [일련의 스레드 덤프 캡처](https://experienceleague.adobe.com/docs/experience-cloud-kcs/kbarticles/KA-17452.html) 및 [분석](https://helpx.adobe.com/kr/experience-manager/kb/thread-dump-analysis.html).
2. AEM Java 프로세스가 높은 CPU 사용률을 초래하는지 OS 수준에서 확인합니다.        AEM에서 CPU 사용률이 높은 경우 몇 분 동안 기본 제공 프로파일링 툴을 실행하고 결과를 분석합니다.
   - Linux: top 명령을 사용하여 CPU 사용률을 확인합니다.
   - 창: windows 작업 관리자 사용
3. [느린 요청에 대해 request.log 파일 분석](https://experienceleague.adobe.com/docs/experience-manager-65/deploying/configuring/monitoring-and-maintaining.html?lang=en#interpreting-the-request-log).
4. 시스템 유지 관리 절차를 검토합니다. 다음 보기 [문서](https://experienceleague.adobe.com/docs/experience-manager-65/deploying/configuring/monitoring-and-maintaining.html)또는 AEM 유지 관리에 대한 자세한 내용을 확인하고 다음을 포함하여 AEM에서 적절한 유지 관리를 수행하는지 확인합니다.
   - 수정 정리(MongoMK 및 Database DocumentNodeStore만 해당) - 매일 또는 더 자주
   - 오프라인 Tar 압축(TarMK만 해당) - 격주
   - 데이터 저장소 가비지 수집(FileDataStore 또는 S3 DataStore만 사용하는 시스템) - 주별
   - 워크플로우 삭제 - 주별
   - 버전 삭제 - 주별
   - 감사 로그 삭제 - 매주
5. 에서 구현된 캐싱 전략 검토 [AEM dispatcher 수준](https://experienceleague.adobe.com/docs/experience-cloud-kcs/kbarticles/KA-17490.html).
6. 사이트의 [캐싱](https://experienceleague.adobe.com/docs/experience-cloud-kcs/kbarticles/KA-17461.html).
7. 와 같은 클라이언트 측 사이트 분석 도구 사용 <b>감사</b> Google Chrome 브라우저의 기능 <b>개발자 도구</b> 패널.  이러한 도구는 클라이언트측 성능 개선에 대한 권장 사항을 제공합니다.


<b>일반적인 성능 문제에 대한 솔루션</b>

- 을(를) 참조하십시오. [이 문서](https://experienceleague.adobe.com/docs/experience-cloud-kcs/kbarticles/KA-17486.html) 를 참조하십시오.
- 를 검토합니다. [성능 조정 팁](https://experienceleague.adobe.com/docs/experience-cloud-kcs/kbarticles/KA-17466.html)

<br> <u><b>자산 성능 문제</b></u><br> 
<b>Assets 성능 문제의 증상</b>

- /assets.html 또는 /damadmin UI에 대한 느린 파일 업로드
- 축소판을 생성하는 데 시간이 너무 오래 걸립니다
- 이동, 삭제, 편집 및 메타데이터 업데이트와 같은 자산 작업이 너무 오래 걸립니다


<b>Assets 성능에 문제가 있는 이유</b>

- 적절한 유지 관리 부족
- 최신 수정 팩이 적용되지 않음
- 최적화가 적용되지 않음
- 사용자 로드에 대한 서버 크기 조정이 부적합


<b>자산 성능 문제를 분석하는 방법</b>

- [위의 사이트 성능 분석에서 1-4단계를 검토하십시오](https://experienceleague.adobe.com/docs/experience-cloud-kcs/kbarticles/KA-17457.html#performance-analysis).

<b>일반적인 자산 성능 문제에 대한 솔루션</b>
- 를 검토합니다. [AEM Assets 성능 조정 가이드](https://experienceleague.adobe.com/docs/experience-manager-65/deploying/configuring/assets-performance-sizing.html?lang=en).
- 일부 문제 시나리오 및 해당 솔루션에 대한 솔루션을 찾을 수 있습니다. [여기](https://helpx.adobe.com/experience-manager/kb/comon-aem-assets-ingestion-issues.html).
- 자산 처리 성능 조정에 대한 자세한 내용은 [이 문서](https://helpx.adobe.com/experience-manager/kb/Asset-Performance-Tuning.html).

<b><u>메모리 문제</u></b>
 <br>
<b>메모리 문제 증상</b>

- AEM이 임의로 충돌하고 OutOfMemoryError 로그에서 관찰됩니다
- AEM은 시간이 지남에 따라 느려지고 결국 충돌이 발생합니다
- AEM이 응답하지 않습니다.


<b>메모리 문제 진단</b>

- 로그 파일에서 OutOfMemoryError를 검색하면 일치하는 항목이 있으면 메모리 문제가 발생합니다
- http:// 를 검토합니다.*aem-host:port*/system/console/memoryusage screen

   &quot;이전 세대&quot;(JDK 7 이전) 또는 &quot;Tested Generation&quot;(JDK8 이상) 사용량이 많은 경우 이는 heap 메모리 사용 문제가 될 수 있습니다.  &quot;가비지 수집기 실행&quot;을 눌러 JVM이 전체 힙의 가비지 수집을 실행하도록 요청합니다.  GC를 요청한 후 높은 heap 활용률이 유지되면 문제가 발생할 수 있습니다.  Oak Tar 저장소가 있는 AEM 인스턴스에서 종래의 사용량이 3GB보다 많으면 문제가 발생할 수 있습니다.  메모리 내 캐시 구성으로 인해 Mongo 스토리지가 있는 시스템의 높은 힙사용률이 발생할 수 있습니다.
- [스레드 덤프 및 상위 출력 가져오기](https://experienceleague.adobe.com/docs/experience-cloud-kcs/kbarticles/KA-17452.html#) 다음을 수행합니다. [스레드 분석](https://helpx.adobe.com/kr/experience-manager/kb/thread-dump-analysis.html).  높은 CPU 사용률을 일으키는 스레드가 기본 JVM 가비지 수집 스레드인지 확인합니다.  CPU 시간이 가장 많은 스레드가 &quot;VM 스레드&quot; 또는 가비지 수집 스레드인 경우 메모리 문제가 발생할 수 있습니다.


<b>메모리 문제의 원인</b>

- Java 응용 프로그램 메모리 누수
- 사용자 지정 코드에서 완료의 잘못된 사용으로 인해 Java Finalizer가 빌드됩니다.
- 최대 heap 구성이 부족합니다.


<b>메모리 문제의 원인을 분석하는 방법</b>

자세한 내용은 [이 문서](https://experienceleague.adobe.com/docs/experience-cloud-kcs/kbarticles/KA-17482.html) heap 덤프를 캡처하는 방법에 대한 자세한 내용을 참조하십시오.

메모리 문제의 원인을 식별하는 가장 좋은 방법은 힙용 덤프를 분석하는 것입니다.

Heap 덤프 파일을 캡처한 후 [Eclipse 매트](http://www.eclipse.org/mat/) 또는 [IBM 메모리 분석기](https://developer.ibm.com/languages/java/) 도구.  Eclipse MAT에서 리크 용의자 보고서를 실행하고 &quot;스레드 세부 정보&quot; 보기를 열어 메모리 문제에 대한 잠재적 원인을 확인합니다.

<b>일반적인 메모리 문제에 대한 솔루션</b>

- 장기간 가비지 수집을 일시 중지하는 경우 메모리를 적게 사용하도록 애플리케이션 코드를 최적화합니다.  대부분의 가비지 수집 문제는 응용 프로그램을 최적화하는 것과 JVM을 조정함으로써 해결되는 것이 가장 좋습니다.
- 이미 응용 프로그램을 최적화했지만 GC가 긴 경우 일시 중지됩니다. [JVM 조정에 중점을 둡니다.](https://docs.oracle.com/javase/8/docs/technotes/guides/vm/gctuning/).

<u><b>AEM 색인 지정 문제</b></u>
 <br>
<b>인덱싱 문제 증상</b>

다음은 AEM/Oak 색인 지정 시 발생하는 문제의 표시입니다.

- 검색 결과가 10분 이상 오래된 것입니다
- 검색 결과가 없습니다.
- 사이트 UI, Query Builder 검색 또는 JCR 쿼리 실행을 통해 검색하는 동안 UI에서 오류가 반환되거나 로그가 반환됩니다

<b>인덱싱 문제 진단</b>
비동기 인덱싱이 느리거나 실패하는지 확인하려면 다음을 수행하십시오.
1. AEM 인스턴스에서 다음 URL을 열어 비동기 인덱서에 대한 통계를 확인합니다. http://aemhost:port/system/console/jmx/org.apache.jackrabbit.oak%3Aname%3Dasync%2Ctype%3DIndexStats http://aemhost:port/system/console/jmx/org.apache.jackrabbit.oak%3Aname%3Dfulltext-async%2Ctype%3DIndexStats  - This URL은 AEM6.2 이상에만 적용됩니다
2. 각 페이지에서 다음 필드를 확인합니다.

   <b>FailureSince</b> - 색인이 처음 실패하는 경우를 나타냅니다.

   <b>마지막 오류</b> - 색인화가 실패하는 원인을 보여주는 스택 추적입니다.  비어 있는 경우 색인화가 실패하지 않습니다.

   <b>LastErrorTime</b> - 색인화가 마지막으로 오류를 발생했음을 나타냅니다.

   <b>LastIndexedTime</b> - 이 필드의 날짜 및 시간이 5분을 초과하는 경우 색인이 너무 느리게 실행됩니다.


<b>색인 지정 시 문제가 발생하는 이유</b>

- 수정 가비지 수집, 워크플로우 삭제, 감사 삭제, 버전 삭제 등과 같은 유지 관리를 수행하지 않거나 부적절한 유지 관리
- Tar 저장소의 세그먼트가 손상되었거나 누락되었습니다
- 클러스터형 환경의 수정 손상(DocumentNodeStore - Mongo 또는 데이터베이스)
- 클러스터 환경의 클러스터 토폴로지 문제


<b>인덱싱 문제의 원인이 되는 항목을 분석하는 방법</b>

- 자세한 내용은 [이 문서](https://experienceleague.adobe.com/docs/experience-cloud-kcs/kbarticles/KA-17492.html) 인덱싱 문제 분석 및 수정

<b><u>복제 문제</u></b>
 <br>
<b>복제 문제 증상</b>

- 게시 요청이 복제 에이전트 큐에 대기 중입니다.
- 게시된 내용이 게시 서버에 표시되지 않습니다.
- 시스템 성능에 미치는 영향


<b>복제 문제의 원인:</b>

- 복제 에이전트가 잘못 구성되어 게시 에이전트에 연결할 수 없습니다
- 복제 시 오류가 발생하여 복제 큐가 중지됩니다.
- 시스템이 느리고 복제 처리 속도가 느려집니다.
- 복제가 사용자 지정 워크플로우의 일부로서 발생하고 있으며 워크플로우 처리에 문제가 있습니다.


<b>복제 문제를 분석하는 방법:</b>

1. 복제 큐 확인 [상태](https://experienceleague.adobe.com/docs/experience-manager-65/deploying/configuring/replication.html?lang=ko-KR):

   <b>활성:</b> 항목을 처리하는 경우

   <b>유휴 상태:</b> 큐가 비어 있는 경우

   <b>차단됨:</b> 항목이 큐에 있지만 처리할 수 없는 경우 예를 들어 에이전트가 작동이 중지되었거나 존재하지 않는 호스트를 가리키는 경우입니다.
2. 서버가 복제되었거나 에이전트가 최근에 구성된 경우 복제 구성을 검토하십시오. 자세한 내용은 [여기](https://experienceleague.adobe.com/docs/experience-manager-65/deploying/configuring/replication.html?lang=en#configuring-your-replication-agents-from-the-author-environment).
3. 에서 복제 에이전트 로그를 검토합니다. <b>http://host:port/etc/replication/agents.author/AgentName.log.html#end</b>.  항목을 식별할 수 없는 경우 이 로그를 수집하여 AEM 지원에 문의하십시오.
4. 다음 위치에서 서버 error.log 검토 <b>AEM 설치/crx-quickstart/logs</b>; 항목을 식별할 수 없는 경우 이 로그를 수집하여 AEM 지원에 문의하십시오.
5. 복제 큐가 &quot;유휴&quot; 상태이고 위의 항목이 적용되지 않는 경우 이 경우 워크플로우로 인해 문제가 발생할 수 있습니다. 워크플로우가 처리되지 않는 경우 복제 항목이 복제 큐에 도달하지 않습니다. 워크플로우의 상태를 모니터링하려면 워크플로우 대시보드를 확인하여 실행 중인 워크플로우 인스턴스 수를 확인할 수 있습니다. 워크플로우 관리에 대해 읽을 수 있습니다 [여기](https://experienceleague.adobe.com/docs/experience-manager-65/administering/operations/workflows.html).
6. 시스템의 로드가 높거나 다른 성능 문제가 발생할 경우 복제 속도가 느려집니다.


<b>일반적인 복제 문제에 대한 해결 방법:</b>

1. [복제 큐 문제 검토](https://experienceleague.adobe.com/docs/experience-cloud-kcs/kbarticles/KA-17467.html).
2. 효율적으로 실행되지 않는 워크플로우로 인해 문제가 발생하는 경우 동시 실행 항목을 검토할 수 있습니다 [워크플로우 처리 팁](https://experienceleague.adobe.com/docs/experience-manager-65/developing/extending-aem/extending-workflows/workflows-process-ref.html?lang=en).

<b><u>TarMK 손상 문제</u></b>
 <br>
<b>TarMK 손상 증상</b>

- 오프라인 압축 후 인스턴스가 작동하지 않습니다.
- 인스턴스가 다음 위치에 있습니다. <b>시작 진행 중</b> state.
- 로그 파일 또는 압축 명령 출력 보고서 <b>SegmentNotFoundException</b>.


<b>손상 문제의 원인</b>

- 세그먼트는 수동 개입(예: rm -rf )으로 제거됩니다.
- 세그먼트가 수정 가비지 수집에 의해 제거되거나 코드의 일부 버그로 인해 세그먼트를 찾을 수 없습니다.
- 코드의 일부 버그로 인해 세그먼트를 찾을 수 없습니다.
- 다양한 유지 관리 작업이 제시간에 수행되지 않아 저장소 증가 및 디스크 공간이 줄어듭니다.
- Java 프로세스를 종료하여 AEM을 강제로 중지합니다.


<b>저장소 손상 문제 진단:</b>

- error.log 파일을 검토하고 있는지 확인합니다. <b>SegmentNotFoundException</b> 또는 <b>잘못된 인수 예외</b>.
- 세그먼트가 수정 가비지 수집에 의해 제거되었는지 확인하려면 org.apache.jackrabbit.oak.plugins.segment.file.TarReader-GC(디버그 로그 사용) 로거의 출력을 확인하십시오. 해당 로거는 정리 단계에서 제거된 모든 세그먼트의 세그먼트 ID를 기록합니다. 해당 로거의 출력에 잘못된 세그먼트 ID가 표시되는 경우에만 수정 가비지 수집이 예외의 원인입니다.
- 외부 데이터 저장소가 손상된 경우 로그 파일에서 모든 오류 발생 여부를 검색합니다 <b>blobId에 대한 InputStream을 가져오는 동안 오류가 발생했습니다.</b>. 이 오류는 AEM 데이터 저장소 디렉터리에서 파일이 없음을 의미합니다.


<b>손상 문제를 복구하는 해결 방법:</b>

- 를 사용하여 세그먼트 저장소의 마지막으로 알려진 양호한 수정 버전을 확인합니다. [check](https://github.com/apache/jackrabbit-oak/blob/trunk/oak-run/README.md#check) oak-run의 실행 모드.  손상된 세그먼트 저장소를 수동으로 최근의 양호한 수정 버전으로 되돌립니다. 이 작업은 Oak 리포지토리를 제 시간에 이전 상태로 되돌립니다.  이 작업을 수행하기 전에 리포지토리를 완전히 백업해야 합니다.
   - 확인 및 복원을 수행하려면 다음에 언급된 단계를 수행합니다. [이 문서](https://experienceleague.adobe.com/docs/experience-cloud-kcs/kbarticles/KA-16542.html).
   - 검사에 실패하면 <b>ConsistencyChecker - 양호한 수정 버전이 없음</b> 그런 다음 [이 문서](https://helpx.adobe.com/experience-manager/kb/offline-compaction-fails-with-SegmentNotFoundException-and-IllegalArgumentException.html).
- 데이터 저장소를 사용하지 않는 경우 기본 세그먼트 저장소 대신 외부 파일, S3 또는 Azure 데이터 저장소를 사용합니다.
   - 데이터 저장소를 사용하면 성능이 향상됩니다.
   - 을 사용하여 인스턴스를 데이터 스토어로 마이그레이션 [crx2oak](https://experienceleague.adobe.com/docs/experience-manager-65/deploying/upgrading/using-crx2oak.html?lang=en).
- 최신 서비스 팩 및 누적 수정 팩 및 Oak 누적 수정 팩을 적용합니다.
