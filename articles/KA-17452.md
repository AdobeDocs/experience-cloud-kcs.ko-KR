# JVM에서 스레드 덤프 가져오기

## 설명

<br><br><br>JVM에서 스레드 덤프를 가져오려면 어떻게 해야 합니까? [!DNL Linux], UNIX 또는 [!DNL Windows]?<br><br><br><br> <br><br>
스레드 덤프는 [!DNL Java] 현재 [!DNL Java Virtual Machine] (JVM).

JVM에서 스레드 덤프를 가져오는 방법에는 여러 가지가 있습니다. 스레드 덤프를 2개 이상 사용하는 것이 좋습니다. 일반적인 간격에 10개의 스레드 덤프를 사용하는 것이 좋습니다(예: 10초마다 한 개의 스레드 덤프).


## 해상도

<br><br>1단계: PID를 가져옵니다 [!DNL Java] 프로세스<br><br><br><br> <br><br>
스레드 덤프를 가져올 수 있는 첫 번째 정보는 [!DNL Java] 프로세스의 PID입니다.

다음 [!DNL Java] JDK는 모든 파일을 나열하는 jps 명령과 함께 제공됩니다 [!DNL Java] 프로세스 id입니다. 다음과 같이 이 명령을 실행할 수 있습니다.


```
jps -l 70660 sun.tools.jps.Jps 70305
```


<b>참고:</b> in [!DNL Linux] 그리고 UNIX에서는 이 명령을 `sudo -u user jps -l`이고 여기서 &quot;user&quot;는 [!DNL Java] 프로세스 실행 중

이 방법이 작동하지 않거나 여전히 을(를) 찾을 수 없는 경우 [!DNL Java] 프로세스, (경로가 설정되지 않음, JDK가 설치되지 않음 또는 이전 [!DNL Java] version),

- UNIX, [!DNL Linux], 및 Mac OS X: `ps -el | grep java`
- [!DNL Windows]: Ctrl+Shift+Esc 키를 눌러 작업 관리자를 열고 의 PID를 찾습니다 [!DNL Java] 프로세스

<br><br><br><br><br><br>2단계: JVM에서 스레드 덤프 요청<br><br><br><br><br><br><br><br>
<b>jstack</b>

설치/사용 가능한 경우 <b>jstack</b> 도구. 명령줄 콘솔에 스레드 덤프를 인쇄합니다.

jstack을 사용하여 스레드 덤프를 가져오려면 다음 명령을 실행합니다.
`jstack -l pid`

콘솔 출력 redirect/append 지시어를 사용하여 연속 스레드 덤프를 파일에 출력할 수 있습니다.
`jstack -l pid  threaddumps.log`

참고:

- jstack 도구는 JDK 1.5 이후 사용할 수 있습니다(JVM용) [!DNL Windows], 일부 버전의 JDK 1.5 및 JDK 1.6에서만 사용할 수 있습니다.)
- jstack은 `-Xrs` jvm 매개 변수가 활성화됨
- JDK 1.5에서 실행 중인 프로세스에서 JDK 1.6의 jstack 도구를 사용하여 덤프 작업을 수행할 수 없습니다.
- in [!DNL Linux] 및 UNIX에서는 java 프로세스를 소유하는 사용자로 명령을 실행해야 합니다.

   `sudo -u *java-user* jstack -l *pid*`  

   (java-user 는 [!DNL Java] 프로세스는 (으)로 실행됩니다.)
- in [!DNL Windows]jstack을 실행하고 &quot;이 명령을 처리할 수 있는 저장 공간이 부족합니다&quot; 오류가 발생하면 jstack을 windows SYSTEM 사용자 또는 java 프로세스를 소유하는 사용자로 실행해야 합니다.  다운로드할 수 있는 Psexec를 사용하여 이 작업을 수행할 수 있습니다 [여기](http://technet.microsoft.com/en-us/sysinternals/bb897553.aspx). jstack을 SYSTEM 사용자로 실행하려면 다음과 같은 명령을 사용합니다.

   `psexec -s jstack *pid*   threaddumps.log`

   서버에 psexec를 설치할 수 없는 경우 명령이 포함된 .bat 파일을 만들어 실행할 수 있습니다 [사용 [!DNL Windows] 작업 스케줄러(다른 사용자)](https://technet.microsoft.com/en-us/library/cc748993%28v=ws.11%29.aspx).
- Java 프로세스가 응답하지 않는 경우 옵션을 사용하는 데 도움이 될 수 있습니다 <b>`-J-d64`</b> (64비트 시스템의 경우) 예를 들면 다음과 같습니다.

   `jstack -J-d64 -l *pid*  threaddumps.log`
- jstack 명령(`jstack -l pid  threaddumps.log`)에서 아래의 오류 1을 발생시킨 다음 java 프로세스를 소유하는 사용자로 명령을 실행합니다.  예:

   `sudo -u sling jstack -l pid  threaddumps.log`

<br><br>JSTACK 스크립트<br><br>
여기 있습니다 [script](https://github.com/cqsupport/jstackSeries.sh/blob/master/jstackSeries.sh) (1990년 10월 10일에 [eclipse.org](http://wiki.eclipse.org/How_to_report_a_deadlock#jstackSeries_--_jstack_sampling_in_fixed_time_intervals_.28tested_on_Linux.29)) jstack을 사용하여 일련의 스레드 덤프를 가져옵니다.  또한 top 명령을 사용하여 스레드 수준 cpu 사용도 수행합니다.

다음과 같이 실행합니다.
`sudo -u *user*jstackSeries.sh *pid* *aemserveruser* *count* *delay*`

예:
`sudo -u aemuser jstackSeries.sh 1234 aemserveruser 10 3`

- <b>1234년</b> 는 의 pid입니다 [!DNL Java] 프로세스
- <b>cq5serveruser</b> 은 [!DNL Linux] 또는 [!DNL Java] 다음으로 실행
- <b>10</b> 가져올 스레드 덤프 수
- <b>3</b> 각 덤프 사이의 지연 시간


<b>참고: </b>위쪽 출력에는 기본 스레드 ID가 16진수로 표시되는 반면 jstack 출력은 16진수로 표시됩니다.  스레드 ID를 16진수로 변환하여 맨 위 출력의 높은 CPU 스레드를 jstack 출력에 일치시킬 수 있습니다.

위의 스크립트 외에 유사한 기능이 있습니다 [[!DNL Windows] github의 Powershell 스크립트 및 Adobe AEM 관련 스크립트](https://github.com/cqsupport/jstackSeries.sh).

<b>Adobe Experience Manager용 스레드 덤프 도구</b>

Adobe Experience Manager 제품을 사용 중이라면 설치할 수 있습니다 [이 도구](https://helpx.adobe.com/experience-manager/kb/thread-dumps-collection-analysis.html) 스레드 덤프를 생성하기 위한 간단한 UI가 있어야 합니다.
<br><br>스레드 덤프를 가져오는 다른 방법<br><br>
만약 <b>jstack</b> 도구를 사용할 수 없으면 다음과 같이 스레드 덤프를 가져올 수 있습니다.

<b>참고:</b> 일부 도구는 명령줄 매개 변수가 있는 경우 JVM에서 스레드 덤프를 가져올 수 없습니다 `-Xrs` 이 활성화되어 있습니다. 스레드 덤프를 가져오는 데 문제가 있는 경우 이 옵션이 활성화되어 있는지 확인하십시오.
<br><br>UNIX, Mac OS X 및 [!DNL Linux] (JDK 1.4 이하 버전)<br><br>
UNIX의 경우 Mac OS X 및 [!DNL Linux]로 QUIT 신호를 보낼 수 있습니다 [!DNL Java] 스레드 덤프를 표준 출력으로 출력하도록 하는 프로세스입니다.

1. 이 명령을 실행하여 다음을 수행합니다.

   `kill -QUIT *pid*`

   이 명령을 다음과 같이 실행해야 할 수 있습니다. `sudo -u user kill -QUIT *pid*` 여기서 &quot;user&quot;는 [!DNL Java] 프로세스 실행 중
2. 를 사용하여 CQSE를 시작하는 경우 `crx-quickstart/server/start` 스크립트를 작성한 후 스레드 덤프가 `crx-quickstart/server/logs/startup.log`. JBoss와 같은 타사 애플리케이션 서버를 사용하는 경우, [!DNL WebSphere], [!DNL Tomcat]또는 서버의 설명서를 참조하여 표준 출력이 어디로 이동되는지 확인하십시오.

[!DNL Windows]:<br><br>JDK 1.X
1. javadump.exe (아래에 첨부됨)를 다운로드합니다.
2. 다음 세 개의 인수로 JVM을 시작합니다(올바른 순서로 지정되어야 함).

   `-XX:+UnlockDiagnosticVMOptions -XX:+LogVMOutput -XX:LogFile=*C:\temp\jvmoutput.log*`
3. 작업 관리자를 열려면 Ctrl+Shift+Esc 키를 누릅니다.
4. 의 PID를 찾습니다 [!DNL Java] 프로세스.
5. 명령줄에서

   `javadump.exe *pid*`
6. 스레드 덤프가 `jvmoutput.log` 2단계에서 언급한 파일

<br><br>JDK 1.6<br><br>
스레드 덤프 가져오기 <b>jconsole</b> 도구, 플러그인 사용: 0

스레드 덤프를 요청하는 방법은 다음과 같습니다.

1. 실행 중인 jvm에 다음 매개 변수를 추가합니다 `Communique` : `-Dcom.sun.management.jmxremote`
2. JDK 1.6(아직 다운로드하지 않은 경우)을 다운로드하여 설치합니다.
3. 다운로드 및 추출 [스레드 덤프 분석기 유틸리티](https://github.com/irockel/tda). 1
4. 실행 `jconsole.exe` JDK 1.6:

                               `jconsole.exe -pluginpath /path/to/file/tda.jar`
5. 을(를) 클릭합니다. <b>스레드 덤프</b> 탭.
6. 을(를) 클릭합니다. <b>요청 스레드 덤프</b> 링크를 클릭합니다.

<b>참고:</b> AEM 6을 실행 중인 경우.\* 실행 중인 스레드를 관찰하려면 다음을 요청할 수 있습니다. `http://host:port/system/console/status-Threads` 스레드 목록을 가져옵니다. 그러나 이러한 스레드 덤프는 사무라이 또는 tda와 같은 스레드 덤프 분석 도구에서 작동하지 않습니다.<br><br>적용 대상<br><br>
